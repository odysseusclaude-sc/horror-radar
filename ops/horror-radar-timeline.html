01
Phase I · Origin
Concept & Initial Foundation
The project begins as a personal pain-point solution: finding quality indie horror games is fragmented and noisy. A data-driven discovery platform is conceived.
Founding Premise
The Discovery Problem
The core insight driving the project: the indie horror games space is prolific but opaque. Steam's algorithm is optimized for revenue, not quality. YouTube coverage is dominated by big titles. Reddit is noisy and ephemeral. A dedicated, scored, curated radar could fill a structural gap in the ecosystem — surfacing overlooked games before they peak or disappear entirely.
Initial Scope
Platform Goals Defined
Early decisions establish the platform's three core promises: discovery (surface games before they go viral), scoring (a proprietary quality signal), and curation (a weekly editorial spotlight). These three pillars remain consistent throughout all subsequent phases.
Tool Philosophy
Plan-Before-Code Workflow Adopted
From the outset, a plan-before-code discipline is established. No feature or component is built without first designing its structure, defining its data shape, and articulating its purpose. This discipline shapes every subsequent technical decision and prevents scope creep during build phases.
02
Phase II · Backend
Core Architecture Build
The backend stack is chosen, scaffolded, and integrated with primary data sources. Foundational decisions are made that determine the system's long-term shape.
Stack Decision
Python / FastAPI Selected
FastAPI is chosen as the web framework for its async-native design, automatic OpenAPI schema generation, and Pythonic developer ergonomics. The async model is deemed essential given the platform's need to make concurrent calls to multiple external APIs (Steam, YouTube, Reddit) without blocking.
ORM Layer
SQLAlchemy for Data Persistence
SQLAlchemy is adopted as the ORM. The declarative model approach allows game, score, and metadata entities to be defined clearly. The decision also future-proofs the system: the ORM abstraction makes swapping from SQLite (dev) to Postgres (production) a config-level concern rather than a code rewrite.
Scheduling
APScheduler for Pipeline Automation
APScheduler is integrated to run the data ingestion and scoring pipeline on a defined schedule. This allows the platform to run autonomously — pulling fresh Steam review counts, YouTube video metrics, and Reddit mentions on a recurring basis without manual triggering. The scheduler is embedded within the FastAPI app rather than as a separate service, keeping deployment simple.
Integration
Steam API Integration
The Steam API becomes the primary data source. Key signals extracted: review counts, review sentiment ratios, recent vs. all-time review velocity, release dates, tags, and price. Review velocity (rate of recent reviews vs. all-time) becomes one of the earliest candidate signals for detecting games "breaking out."
Integration
YouTube API Integration
The YouTube Data API is added to capture content creator activity around each game. View counts, video counts, and publication dates around a game's title are pulled and normalised. This layer is designed to detect when a game is getting creator attention before it spikes in Steam reviews — enabling earlier discovery signals.
Integration
Reddit as Social Sentiment Layer
Reddit data is introduced as a third signal layer. Subreddit post activity, upvote velocity, and mention frequency across horror and indie gaming communities are tracked. Reddit tends to be the earliest social signal — discussions happen here before YouTube videos and Steam review waves materialize.
03
Phase III · Intelligence
The OPS Scoring System
Raw data is not enough. A proprietary scoring model is designed to synthesize multi-source signals into a single, interpretable quality and momentum score per game.
Core Model
OPS — Overall Performance Score
The OPS (Overall Performance Score) is the platform's central intellectual contribution. It synthesizes Steam review sentiment and velocity, YouTube video coverage and view momentum, and Reddit community activity into a single normalized score. The design goal is to make OPS function as a signal of emerging quality — not just current popularity. A game with few reviews but accelerating momentum should score competitively against a game with high total reviews but flat growth.
Score Design
Momentum vs. Magnitude Distinction
A critical scoring design decision: the OPS model distinguishes between magnitude (total size of signals) and momentum (rate of change). Large established games dominate magnitude metrics. The platform's value lies in weighting momentum — so small games gaining fast traction surface high despite low absolute numbers.
Taxonomy
Hidden Gems Classification
A dedicated "Hidden Gems" classification tier is developed within the scoring system. Games that score above a quality threshold but fall below a visibility threshold — i.e., high OPS but low total Steam review count and low YouTube view count — are flagged as Hidden Gems. This becomes one of the platform's hero features.
Pipeline
Scheduled Re-Scoring Architecture
The scoring pipeline is designed to run on a cadence, not just on ingest. This means every game in the database is re-scored periodically against fresh signal data. The score history is preserved, enabling timeline intelligence — showing how a game's OPS has evolved over time. This unlocks the game timeline page as a distinct product surface.
04
Phase IV · Interface
UI/UX Design System
With data architecture stable, the focus shifts to surfaces. A multi-tool design process is used — prototyping across Google Stitch, Figma Make, and Figma MCP — guided by Don Norman's design principles.
Design Fixture
Hollowfield as the Mock Game
Hollowfield — a fictional indie horror game — is created as a consistent design fixture. Every UI prompt, mockup, and component uses Hollowfield as the content stand-in. This discipline keeps design work grounded in realistic content shapes (title length, description tone, screenshot composition) and produces cohesive mockups across all sessions.
Framework
Don Norman's Principles as Design Law
Don Norman's principles from The Design of Everyday Things — affordance, feedback, constraints, mapping, and discoverability — are explicitly adopted as the design evaluation framework. Every UI component is judged against these principles. This keeps design decisions principled rather than taste-driven, and creates a shared vocabulary for evaluating mockup quality.
Toolchain
Multi-Tool Design Prototyping
Design exploration uses a parallel toolchain: Claude Code for rapid component generation, Google Stitch for early layout ideation, Figma Make for structured mockup work, and the Figma MCP integration for closing the loop between AI-generated concepts and production Figma files. Multiple design concepts are generated before any single direction is committed to.
Page Design
Hidden Gems Insights Page
The Hidden Gems Insights Page is designed as a dashboard-style surface showing the current cohort of high-OPS, low-visibility games. Design focus: making the score and its components scannable at a glance, with visual hierarchy that emphasizes momentum signals. Card-based layout with sortable/filterable state.
Page Design
Game Timeline Intelligence Page
The Game Timeline Intelligence Page surfaces the historical OPS trajectory for a single game. Design challenge: communicating score change over time in a way that's immediately interpretable without being data-heavy. The page also overlays external events (YouTube video publication, Reddit spike dates) onto the OPS curve — making the cause-effect relationship visible.
Page Design
Radar Pick Spotlight Page
The Radar Pick Page is the platform's editorial crown jewel — a single-game deep-dive published weekly. Design emphasis: magazine-quality editorial layout, long-form reading experience, strong visual identity for the selected game. Hollowfield was the prototype subject. The page is intentionally distinct from the data-dense discovery pages — it is slow, immersive, and opinionated.
05
Phase V · Strategy
Market Analysis & Product Positioning
With core product in view, a serious market analysis phase begins. Who is this for? What is its competitive position? A full buyer persona analysis leads to a critical strategic pivot.
Research
Indie Horror Games Market Analysis
A structured market analysis of the indie horror games ecosystem is conducted — covering market size, release volume trends, Steam discoverability dynamics, the role of YouTube in driving indie game sales, and the competitive landscape of existing discovery tools. Named references and real data sources are used rather than generalizations.
Personas
Buyer Persona Development
Multiple buyer personas are developed and evaluated: casual horror game fans, hardcore genre enthusiasts, indie game developers wanting visibility benchmarks, and content creators seeking games to cover. Each persona is assessed for willingness to pay, frequency of platform use, and strategic fit with the platform's data capabilities.
Pivot Point
Primary Target: Mid-Tier YouTube Creators
Mid-tier YouTube horror game creators (channels with ~10K–500K subscribers) are identified as the primary target persona. The reasoning: they have a professional need to find quality games early (before their competitors), a concrete monetary value in being first to cover a breakout title, limited time to do manual research across Steam/Reddit/YouTube, and enough audience scale to benefit meaningfully from OPS-driven discovery. This persona shapes all subsequent product decisions.
Strategic Reframe
From SaaS Product to Personal Curation Platform
The most consequential strategic pivot: the platform is reframed away from a standalone SaaS product (with external users as customers) toward being a personal curation infrastructure supporting the builder's own emerging creator identity. The platform is the engine behind a horror curator brand — generating credibility, discovery data, and weekly editorial content — rather than a product sold independently. This reframe unifies the technical and creative dimensions of the project under a single mission.
06
Phase VI · Brand
Creator Identity & Content Strategy
With the platform reframed as personal infrastructure, a creator strategy is built to match. Positioning, platform sequencing, and publication cadence are all designed from first principles.
Positioning
Cross-Media Horror Curator Identity
The creator identity is defined as a cross-media horror curator — covering games, film, books, and art through the lens of genre expertise. This breadth distinguishes the brand from single-medium horror commentators and reflects the builder's genuine, longstanding passion across horror media forms.
Competitive Positioning
Contrasted with Jacob Geller's Style
The creator positioning is explicitly differentiated from Jacob Geller's retrospective essay approach — deep, slow, literary video essays on established titles. The Radar's identity is positioned as forward-looking and discovery-oriented: finding what's emerging, scoring it, and creating the record of what was seen before it was obvious. Present-tense curation vs. retrospective analysis.
Roadmap
3-Month Creator Launch Playbook
A sequenced 3-month content launch playbook is designed: begin with Substack for written discovery dispatches (low friction, builds a reading audience), launch Discord in parallel for community around the curation brand, and move to YouTube in the third phase once the editorial voice is established in text and there is an engaged community to seed the channel's early growth.
Content Architecture
Radar Pick as Weekly Anchor Content
The weekly Radar Pick — the editorial spotlight page on the platform — becomes the anchor content unit. Each week, one game is selected, scored, and written up in depth. This provides the Substack with a structured publication cadence, the Discord with a weekly discussion prompt, and eventually YouTube with a scripted subject. One piece of research, three platform outputs.
07
Phase VII · Frontier
AI Agent Orchestration Exploration
The final frontier: exploring how AI agent architectures can augment or automate layers of the platform — from data analysis to content generation to autonomous pipeline management.
Exploration
AI Agents as Platform Augmentation
Interest in AI agent orchestration emerges as a way to go beyond scheduled scripts. The hypothesis: AI agents could analyze incoming game data, flag anomalies in OPS trajectories, draft preliminary Radar Pick write-ups, or even suggest score weight adjustments based on observed signal patterns. This represents a shift from the platform as a data pipeline toward the platform as an intelligent system.
Evaluation
LLM Evaluation Methods Research
In parallel with agent exploration, a deep dive into LLM evaluation methodology is conducted — covering human evaluation frameworks, automated metrics (BLEU, ROUGE, BERTScore), LLM-as-judge approaches, and standard benchmarks (MMLU, HumanEval, SWE-bench). The motivation: any AI-generated content layer in the platform requires rigorous quality assessment before it can be trusted in production editorial output.
Current State
Where the Project Stands Now
The Horror Indie Game Radar exists at a rich convergence point: a functioning data backend (FastAPI + SQLAlchemy + APScheduler, three API integrations, OPS scoring), a designed product UI (three key pages prototyped, design system in progress, Figma integration active), a clear strategic identity (personal curation infrastructure, mid-tier creator audience, cross-media horror brand), and an emerging AI augmentation layer being explored. The platform is simultaneously a technical project, an editorial product, and a creator brand in formation.
